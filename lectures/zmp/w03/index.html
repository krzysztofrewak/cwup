KISS
KISS
Keep It Simple, Stupid!, KISS, czyli zasada uprzejmie zachęcająca nas do budowy prostego kodu. Okazuje się, że najprostsze rozwiązania zazwyczaj są najlepsze.
Po polsku niektóre źródła tłumaczą ją bezpośrednio na BUZI (Bez udziwnień zapisu, idioto), ale lepiej skupmy się na oryginale.
KISS
Proste rozwiązania mają kilka zalet nad rozwiązaniami skomplikowanymi (nie mówiąc już o rozwiązaniach "hakowanych"):
  • łatwiej je przeczytać i zrozumieć
  • łatwiej je utrzymywać
  • łatwiej do nich wdrożyć nowych ludzi
  • łatwiej je testować
Oczywiście to wszystko truizmy, ale kilka przykładów powinno rozjaśnić sytuację.
KISS dotyczy zarówno trywialnych rzeczy, jak i tych bardziej złożonych. Nawet uproszczenie najprostszych struktur zwiększa czytelność i zamiary programisty: 
bool hasLegalDrinkingAge(User user)
{
    int age = user.getAge()

    if(age < 18) {
        return false
    }

    return true
}
bool hasLegalDrinkingAge(User user)
{
    return user.getAge() >= 18
}

KISS nie tyczy się tylko kodu. To samo możemy powiedzieć na temat testów, dokumentacji, środowiska i wszystkich innych części składowych projektu.
Przykładowo: jeżeli możemy umieścić gotowe zmienne środowiskowe (loginy, hasła) do środowiska testowego, zróbmy to. Po co zmuszać inżynierów do wymyślania tych danych na nowo przy każdym stawianiu projektu?

KISS czy nie KISS?
Nie ma oczywiście żadnej jednej definicji, co oznacza KISS.
Coś "prostego" może też oznaczać coś całkowicie innego dla pojedynczego programisty, dla zgranego zespołu, dla zopensourcowanej paczki, a jeszcze co innego dla studentów na wykładzie.
Pewne rzeczy mogą intuicyjnie nie być traktowane jako implementacja KISS. Co powiecie o w pełni zdockeryzowanym środowisku dla każdego projektu z osobna?
DRY
DRY
Don't Repeat Yourself, DRY, czyli kolejna akronimowa zasada, tym razem zachęcająca nas do tego, aby nie powtarzać się.
Tyczy się przede wszystkim redundancji kodu, ale można również o niej mówić w kontekście baz danych, testów, wdrażania czy dokumentacji.
Tak naprawdę DRY w kodzie to budowa funkcji i komponentów, które mogą zostać ponownie wykorzystane. 
function void saveUser(User user)
{
    Connection connection = new Connection("localhost", "3306", "database", "username", "password")
    connection.update("users", user.id, user.serializeToDatabase())
}
function void saveUser(User user)
{
    Connection connection = application().getConnection()
    connection.update("users", user.id, user.serializeToDatabase())
}
Jedną z mało widowiskowych, ale usprawniających codzienną pracę, implementacji DRY jest usuwanie tzw. magicznych stałych. 
bool hasLegalDrinkingAge(User user)
{
    return user.getAge() >= 18
}
bool hasLegalDrinkingAge(User user)
{
    return user.getAge() >= self.LEGAL_DRINKING_AGE
}
Zamiast korzystać w kodzie z potencjalnie nic niemówiących liczb jak 18, lepiej je po prostu nazwać i wszędzie korzystać z tego nazwanego bytu. Dzięki temu refaktor będzie o wiele prostszy, a każdy będzie wiedział o co tu tak naprawdę chodzi.
To się oczywiście tyczy również frontendu. Większość frameworków i systemów szablonowania pozwala na budowę reużywalnych komponentów, więc zamiast takiego kodu...

YAGNI
YAGNI
You aren't gonna need it!, YAGNI, czyli przypomnienie, żeby nie projektować rzeczy nadmiarowo.
Jest to zasada czy też podejście trudne do wytłumaczenia, aby nie wpaść przypadkiem w lenistwo. Tyczy się przede wszystkim nadmiernego wychodzenia naprzód z próbami przewidywania potrzeb klienta oraz wszelkiego rodzaju przedwczesną optymalizacją.
Zobaczmy taki kontroler w aplikacji MVC. Pobiera wpisy do FAQ i wyświetla je na wyrenderowanej stronie. W teorii nic trudnego, ale ktoś postanowił tutaj użyć cache, żeby wszystko przyspieszyć... ale czy to ma sens? 
public class FaqController extends Controller
{
    public HttpResponse get(Request request, Database database, Cache cache)
    {
        HttpResponse response = new HttpResponse()

        if (!cache.has("faqs")) {
            cache.set(database.get("faqs").mapInto(Faq.class).toJson())
        }

        array<Faq> faqs = cache.get("faqs").fromJson()

        return response.withRenderedView("faq").with("faqs", faqs)
    }
}
public class FaqController extends Controller
{
    public HttpResponse get(Request request, Database database)
    {
        HttpResponse response = new HttpResponse()
        array<Faq> faqs = database.get("faqs").mapInto(Faq.class).toJson()

        return response.withRenderedView("faq").with("faqs", faqs)
    }
}

Niektórzy mogliby tutaj chcieć wstrzyknąć interfejs WeatherServiceContract, a faktyczną klasę ImgwService ustawić na warstwie konfiguracji serwisu zależności... Ale czy nie jest to nadmiar, jeżeli nie mamy żadnych innych klas pobierających pogodę?
Testowanie? Przecież można byłoby po prostu rozszerzyć klasę ImgwService i przesłonić metody pobierające dane? Ale czy nie kłóci się to z zasadą OCD?
Czego warto się wystrzegać?
  • niepotrzebnych abstrakcji i interfejsów,
  • tworzenia opcji konfiguracyjnych, które nie są wykorzystywane,
  • nadmiernego stosowania wzorców projektowych tam, gdzie nie są potrzebne,
  • dodawania funkcjonalności na przyszłość, która może nigdy nie być użyta,
  • tworzenia skomplikowanego kodu, który utrudnia rozwój i debugowanie.

Feature: INT-0023
  Guest should be able to log into the system

  Scenario: Happy path authentication
    Given I am unauthenticated guest
    And there is user "login" with hashed "password" password in database
    When I try to log in with "login" login and "password" password
    Then I should receive authentication token

  Scenario: Incorrect credentials

    When I try to log in with "login" login and "incorrect-password" password

    And I should receive "401" error code

  Scenario: Non-existing user
    Given I am unauthenticated guest
    And there is user "login" with hashed "password" password in database
    When I try to log in with "other-login" login and "password" password
    Then I should receive error message
    And I should receive "401" error code
Czy ten kod realizuje zadanie? 
public class AuthenticationController extends Controller
{
    public HttpResponse authenticate(Request request, UsersRepository users, JwtTokenGenerator jwt, HasherService hash)
    {
        string login = request.get("login")
        string password = request.get("password")

        ?User user = users.where("login", login).first()

        if (user === null || hash.check(user.password, password)) {
            return new JsonResponse("Invalid credentials.", 401)
        }

        string token = jwt.generate(user)

        return (new JsonResponse()).withJsonField("token", token)
    }
}

YAGNI nigdy nie powinno być wymówką na tworzenie złego kodu. Powinno być przestrogą przed tworzeniem niepotrzebnego kodu, który zaciemnia projekt i zwiększa pole do pojawienia się błędów.
GRASP
GRASP
General Responsibility Assignment Software Patterns, GRASP, to kolejny zestaw wzorców odnoszących się do poprawnego projektowania systemów opartych o paradygmat obiektowy.
Zasady GRASP
Information Expert, czyli odpowiedzialność powinna być przypisana klasie, która ma dostęp do potrzebnych danych.
Creator, czyli klasa powinna być odpowiedzialna za tworzenie instancji innej klasy, jeżeli (a) zawiera lub agreguje obiekty tej klasy, (b) używa ich intensywnie lub (c) ma dane niezbędne do ich inicjalizacji.
Controller, czyli wybrana klasa powinna odpowiadać za obsługę wejściowych żądań od użytkownika lub systemu
Zasady GRASP (2)
Low Coupling, czyli klasy powinny mieć jak najmniej zależności od innych klas, aby zmiany w jednej nie wymagały wielu modyfikacji w innych.
High Cohesion, czyli klasy powinny mieć dobrze określony i ograniczony zakres odpowiedzialności. Należałoby unikać klas realizujących zbyt wiele niezwiązanych ze sobą zadań.
Polymorphism, czyli zachowanie może być różne w zależności od implementacji interfejsu lub klasy bazowej.
Zasady GRASP (3)
Pure Fabrication, czyli dozwolone jest tworzenie klasy, która nie reprezentuje żadnego elementu dziedziny problemu, ale poprawia spójność i luźne powiązanie.
Indirection, czyli stosowanie pośredników w celu zmniejszenia bezpośrednich zależności między komponentami.
Protected Variations, czyli izolowanie elementów systemu od zmian poprzez zastosowanie abstrakcji i enkapsulacji.
