KISS
KISS
Keep It Simple, Stupid!, KISS, czyli zasada uprzejmie zachęcająca nas do budowy prostego kodu. Okazuje się, że najprostsze rozwiązania zazwyczaj są najlepsze.
Po polsku niektóre źródła tłumaczą ją bezpośrednio na BUZI (Bez udziwnień zapisu, idioto), ale lepiej skupmy się na oryginale.
KISS
Proste rozwiązania mają kilka zalet nad rozwiązaniami skomplikowanymi (nie mówiąc już o rozwiązaniach "hakowanych"):
  • łatwiej je przeczytać i zrozumieć
  • łatwiej je utrzymywać
  • łatwiej do nich wdrożyć nowych ludzi
  • łatwiej je testować
Oczywiście to wszystko truizmy, ale kilka przykładów powinno rozjaśnić sytuację.
KISS dotyczy zarówno trywialnych rzeczy, jak i tych bardziej złożonych. Nawet uproszczenie najprostszych struktur zwiększa czytelność i zamiary programisty: 
bool hasLegalDrinkingAge(User user)
{
    int age = user.getAge()

    if(age < 18) {
        return false
    }

    return true
}
bool hasLegalDrinkingAge(User user)
{
    return user.getAge() >= 18
}
Nawet takie rzeczy jak nazewnictwo klas i zmiennych można upraszczać. Może nie warto wypisywać wszędzie zbędnych informacji? 
interface GradesCalculator
{
    public GradesCalculator addGradesToCalculator(array<Grade> grades)
    public GradesCalculator addGradeToCalculator(Grade grade)
    public GradesCalculator performCalculations()
    public FinalGrade getCalculatedFinalGrade()
}
interface GradesCalculator
{
    public GradesCalculator add(array<Grade> grades)
    public GradesCalculator add(Grade grade)
    public GradesCalculator calculate()
    public FinalGrade getResult()
}
Inna sprawa, że KISS można zastosować też do uproszczenia wykorzystania danej klasy. Jaki jest sens budowania metody getResult(), jeżeli trzeba pamiętać, aby najpierw wywołać calculate() 
interface GradesCalculator
{
    public GradesCalculator add(array<Grade> grades)
    public GradesCalculator add(Grade grade)
    public GradesCalculator calculate()              # co się stanie jak tego nie wywołamy?
    public FinalGrade getResult()                    # wynik będzie nieprawdziwy?
}
interface GradesCalculator
{
    public GradesCalculator add(array<Grade> grades)
    public GradesCalculator add(Grade grade)
    public FinalGrade calculate()
}
KISS nie tyczy się tylko kodu. To samo możemy powiedzieć na temat testów, dokumentacji, środowiska i wszystkich innych części składowych projektu.
Przykładowo: jeżeli możemy umieścić gotowe zmienne środowiskowe (loginy, hasła) do środowiska testowego, zróbmy to. Po co zmuszać inżynierów do wymyślania tych danych na nowo przy każdym stawianiu projektu?
KISS czy nie KISS?
Nie ma oczywiście żadnej jednej definicji, co oznacza KISS.
Coś "prostego" może też oznaczać coś całkowicie innego dla pojedynczego programisty, dla zgranego zespołu, dla zopensourcowanej paczki, a jeszcze co innego dla studentów na wykładzie.
Pewne rzeczy mogą intuicyjnie nie być traktowane jako implementacja KISS. Co powiecie o w pełni zdockeryzowanym środowisku dla każdego projektu z osobna?
DRY
DRY
Don't Repeat Yourself, DRY, czyli kolejna akronimowa zasada, tym razem zachęcająca nas do tego, aby nie powtarzać się.
Tyczy się przede wszystkim redundancji kodu, ale można również o niej mówić w kontekście baz danych, testów, wdrażania czy dokumentacji.
Tak naprawdę DRY w kodzie to budowa funkcji i komponentów, które mogą zostać ponownie wykorzystane. 
function void saveUser(User user)
{
    Connection connection = new Connection("localhost", "3306", "database", "username", "password")
    connection.update("users", user.id, user.serializeToDatabase())
}
function void saveUser(User user)
{
    Connection connection = application().getConnection()
    connection.update("users", user.id, user.serializeToDatabase())
}
To się oczywiście tyczy również frontendu. Większość frameworków i systemów szablonowania pozwala na budowę reużywalnych komponentów, więc zamiast takiego kodu... 
<div class="flex justify-center bg-white/50">
